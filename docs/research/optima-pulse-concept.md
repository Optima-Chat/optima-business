# Optima Pulse — 产品概念文档

> 创建日期：2026-01-27

## 一句话定义

**Optima Pulse 是一个企业决策 Agent，通过 CLI 工具接入企业数据源，用领域 Skills 编码业务知识，让用户以自然语言对话的方式查询数据和讨论决策。**

## 核心理念

实时感知企业脉搏，做出最优决策。

- **Optima** = 最优解
- **Pulse** = 脉搏/脉动

## 不是什么

| 我们不是 | 为什么 |
|----------|--------|
| BI 工具 | 不做 ETL、不建数据仓库、不画仪表盘 |
| 报表系统 | 不做固定维度的定时报表 |
| Text-to-SQL 工具 | SQL 只是底层手段之一，不是产品体验 |
| 通用 AI 助手 | 绑定特定行业领域知识，不是万能聊天机器人 |

## 是什么

一个**懂行业、能拿数据、会讨论**的 AI 决策参谋。

用户体验示例：

```
用户：上周哪个 SKU 退货率最高？
Pulse：运动鞋 SKU-2031，退货率 12.3%，主要原因是尺码偏小（占退货原因 68%）。
       建议：在商品页增加"偏小半码"的提示，参考 SKU-1892 做过类似调整后退货率降了 40%。

用户：跟供应商聊过了，他们说下一批会调整楦型。那我现在库存还够卖多久？
Pulse：按过去 7 天日均销量 43 双，当前库存 312 双，还能卖 7.3 天。
       但如果退货率降到正常水平（~5%），净销量会提升，大概 6 天就会断货。
       要不要现在下补货单？

用户：先别，等新批次到了再说。帮我盯着，库存低于 100 双提醒我。
Pulse：好的，已设置库存预警：SKU-2031 低于 100 双时通知你。
```

关键区别：**不只是返回数据，而是理解业务语境、给建议、能追问、能设置后续动作。**

## 产品架构

```
┌─────────────────────────────────────────────────┐
│  对话层（用户界面）                                │
│  自然语言输入 → Agent 编排 → 结构化回答 + 建议     │
├─────────────────────────────────────────────────┤
│  Agent 层（编排与推理）                            │
│  理解意图 → 选择工具 → 调用数据 → 结合知识 → 回答  │
├──────────────────┬──────────────────────────────┤
│  Skills 层       │  CLI 工具层                    │
│  （领域知识）     │  （数据获取）                   │
│                  │                               │
│  · 库存管理策略   │  · 电商平台 API                │
│  · 定价建议逻辑   │  · 广告平台 API                │
│  · 退货原因分析   │  · ERP/财务系统                │
│  · 补货时机判断   │  · 物流追踪 API                │
│  · 广告ROI评估   │  · 数据库直连                  │
│  · 行业基准数据   │  · Excel/CSV 导入              │
│  · ...           │  · ...                        │
├──────────────────┴──────────────────────────────┤
│  数据源                                          │
│  Shopify | Amazon | Google Ads | Meta Ads        │
│  Stripe | QuickBooks | 物流API | 自有数据库        │
└─────────────────────────────────────────────────┘
```

### 三层设计原则

**CLI 工具层 — 数据获取**
- 每个数据源 = 一个 CLI 工具
- 标准化接口：输入参数 → 返回结构化数据
- 新增数据源 = 写一个新 CLI，无需 ETL 管道
- 类似 MCP Tools 的设计思路

**Skills 层 — 领域知识**
- 编码行业专家的判断逻辑
- 例如："库存天数 < 7 天且趋势上升 → 建议补货"
- 例如："ROAS < 2 且已投放超过 7 天 → 建议暂停该广告组"
- 可插拔：换行业 = 换一套 Skills
- Skills 既包含规则，也包含行业基准数据

**Agent 层 — 编排与推理**
- 理解用户意图
- 决定调用哪些 CLI 工具获取数据
- 决定使用哪些 Skills 来分析和建议
- 支持多轮对话、追问澄清、主动建议
- 通用层，不绑定具体行业

## 与现有产品的关系

Optima Pulse 的架构与现有的 Optima CLI（commerce-cli）天然契合：

| 现有资产 | 在 Pulse 中的角色 |
|----------|------------------|
| commerce-cli 的 16 个模块 95+ 命令 | CLI 工具层（数据获取）的电商部分 |
| 电商运营 AI Agent 案例的业务逻辑 | Skills 层的电商领域知识 |
| AI 选品与数据分析案例 | Skills 层的选品/分析策略 |

## 首个垂直行业：电商

### 为什么选电商

1. Optima 已有电商 Agent 和数据分析案例，技术积累最深
2. 电商商家数据源相对标准化（Shopify/Amazon/广告平台/物流）
3. 决策场景清晰且高频（补货、定价、广告投放、选品）
4. 中小商家缺乏数据分析人力，付费意愿存在
5. 可做成 SaaS 月费模式，不依赖项目制

### 电商场景的核心 Skills

| Skill | 输入 | 输出 |
|-------|------|------|
| 库存健康诊断 | 库存数据 + 销售趋势 | 补货建议 + 预计断货时间 |
| 广告效果评估 | 广告花费 + 转化数据 | ROAS 分析 + 优化建议 |
| 商品表现排名 | 销售 + 退货 + 评价数据 | 商品健康度评分 + 行动建议 |
| 定价策略建议 | 成本 + 竞品价格 + 销量弹性 | 最优价格区间 |
| 退货原因分析 | 退货数据 + 评价文本 | 根因归类 + 改进建议 |
| 现金流预测 | 收入 + 支出 + 账期数据 | 未来 N 天现金流预测 |
| 选品趋势分析 | 市场数据 + 搜索趋势 | 品类机会评估 |

### 需要接入的数据源（CLI 工具）

| 数据源 | 优先级 | 获取方式 |
|--------|--------|----------|
| Shopify | P0 | REST/GraphQL API |

## 商业模式

### 定价方向

| 套餐 | 目标用户 | 包含内容 |
|------|----------|----------|
| Starter | 单店铺小卖家 | 2 个数据源 + 基础 Skills |
| Growth | 多店铺/多平台 | 5+ 数据源 + 全部 Skills + 预警 |
| Enterprise | 品牌商/大卖家 | 无限数据源 + 自定义 Skills + 私有部署 |

### 关键指标

- 以 MRR（月度经常性收入）为核心指标
- 关注留存率（如果 Pulse 对决策真正有帮助，用户不会取消订阅）
- 用户每周主动对话次数 = 产品价值的最直接衡量

## 差异化壁垒

### vs 传统 BI（帆软/Tableau/Power BI）

- 不需要数据工程师建模和维护
- 不需要学习工具操作
- 输出不是"图表"而是"建议"

### vs 通用 AI（ChatGPT/Claude）

- 直连企业真实数据，不是基于用户粘贴的信息
- 内置行业 Skills，不需要用户自己写 prompt
- 能设置预警和后续动作，不是一次性问答

### vs 现有 Chat BI（FineChatBI/Smartbi）

- 轻量级接入（CLI 工具，非 ETL 管道）
- 决策导向而非数据展示导向
- 垂直行业深度（电商 Skills）而非泛行业浅层对话

## 技术风险

| 风险 | 影响 | 缓解方案 |
|------|------|----------|
| Text-to-SQL 准确率不足 | 返回错误数据导致错误决策 | CLI 工具层直接封装业务查询，减少复杂 SQL 生成 |
| 数据源 API 变更/限流 | 数据获取中断 | 多数据源冗余 + 缓存层 |
| Skills 逻辑与实际业务不符 | 建议不准确 | 允许用户自定义/修正 Skills，持续学习 |
| Agent 幻觉 | 编造数据或建议 | 所有数据回答必须可溯源到具体 CLI 调用结果 |

## 调研背景

本概念文档基于智谱AI商业化调研（见 `docs/research/zhipu-ai/`）的以下关键洞察：

- 智谱行业解决方案本质上都是定制项目，缺乏横向可复制性
- 企业端真正的刚需是"用自然语言跟数据对话并辅助决策"
- 现有 AI BI 产品受限于传统 BI 架构，做不好"决策讨论"这一层
- CLI as tools + Skills as domain knowledge + Agent as orchestrator 的架构可以绕开 BI 的历史包袱
